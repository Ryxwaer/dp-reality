# deployment/07 — Flux CD GitOps installation + Kustomizations

## Goal
Make the K3s cluster pull from this Git repo as its source of truth.
Flux CD reconciles `k3s/base/` (+ overlay) on every push, with image
automation tracking new tags from GHCR.

## Thesis references
- `03-architecture-design.tex` §3.4 "GitOps with Flux CD":
  > "Flux CD, a CNCF graduated project, continuously reconciles the
  > cluster state with the declarative configuration stored in a Git
  > repository. […] If manual changes occur in the cluster (drift),
  > Flux reverts them to match the Git-defined state. Failed
  > deployments trigger automatic rollback to the previous
  > known-good state. […] Flux CD's lightweight footprint
  > (approximately 50–100 MB RAM) makes it suitable for the
  > resource-constrained Raspberry Pi 5 node, unlike heavier
  > alternatives such as Argo CD."

## Dependencies
- Deployment tasks 01–04 are merged (Flux needs something to apply).
- Task 05 is publishing images to GHCR (Flux image-automation
  uses GHCR tags).

## Concrete changes

### Bootstrap directory
`flux/` at repo root:
```
flux/
  clusters/
    prod/
      flux-system/             # generated by `flux bootstrap`
      apps.yaml                # Kustomization that points at k3s/overlays/prod
      infra.yaml               # Kustomization that points at flux/infra/prod
      image-policies.yaml      # ImageRepository + ImagePolicy per service
      image-updates.yaml       # ImageUpdateAutomation (commits tag bumps)
      # Deferred to a later phase:
      #   cert-manager.yaml    — only needed if K3s ingress + ACME replaces NPM
      #   kube-router.yaml     — currently installed via `kubectl apply` from
      #                          the upstream URL (see phase 1 runbook)
  infra/
    prod/
      kustomization.yaml       # explicit resource ordering for the infra layer
      namespaces.yaml          # ingress-nginx, monitoring, flagger-system
      sources.yaml             # HelmRepositories for the three charts
      ingress-nginx.yaml       # HelmRelease (ingress-nginx 4.15.x)
      prometheus.yaml          # HelmRelease (kube-prometheus-stack 85.x)
      flagger.yaml             # HelmRelease (flagger 1.43.x + loadtester)
```

The `infra` Kustomization brings up the ingress-nginx + Prometheus +
Flagger stack that backs the progressive-delivery + automatic-rollback
story (see the "Automatic rollback" section below). It reconciles
independently of the application Kustomization so a chart bump
shows up in `flux get kustomizations` as its own diff.

### Bootstrap command (in `k3s/runbook.md`)
```bash
flux bootstrap github \
  --owner=<owner> \
  --repository=dp-reality \
  --branch=main \
  --path=flux/clusters/prod \
  --personal=true \
  --components-extra=image-reflector-controller,image-automation-controller \
  --read-write-key
```

The two extra flags are mandatory for image-automation to work:
`--components-extra` installs the two image-automation controllers
(not included in the default Flux bundle) and `--read-write-key`
provisions a deploy key with write access (so the image-update
controller can push tag-bump commits back to the repo).

This creates `flux/clusters/prod/flux-system/` with the controller
manifests and a `gotk-sync.yaml` pointing at the same path. After
bootstrap, the operator commits the rest of `flux/clusters/prod/`
(see below) and pushes.

### `apps.yaml`
```yaml
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: dp-reality
  namespace: flux-system
spec:
  interval: 5m
  retryInterval: 1m
  timeout: 10m
  path: ./k3s/overlays/prod
  prune: true
  wait: true
  sourceRef:
    kind: GitRepository
    name: flux-system
  targetNamespace: dp-reality
  # SOPS decryption (task 08).
  decryption:
    provider: sops
    secretRef:
      name: sops-age
```

The Kustomization points at `k3s/overlays/prod` (NOT `k3s/base`)
because image-automation rewrites the `newTag:` setters inside
`k3s/overlays/prod/kustomization.yaml`. Targeting `k3s/base` would
leave that file unreconciled — the auto-bumped tags would never
actually apply to the cluster.

### `image-policies.yaml` + `image-updates.yaml`
For each service image, an `ImageRepository` + `ImagePolicy` that
sorts on an extracted epoch suffix:
```yaml
filterTags:
  pattern: '^main-[a-f0-9]+-(?P<ts>[0-9]+)$'
  extract: '$ts'
policy:
  numerical:
    order: asc
```

The CI workflow therefore tags every push as:
- `latest` (mutable convenience tag for ad-hoc pulls),
- `<full-sha>` (immutable, useful for rollback),
- `main-<sha8>-<epoch>` (the Flux-sortable tag).

A literal `sha-*` tag list (as originally written here) is not
sortable by Flux's `numerical` or `semver` policies — a sortable
suffix is required. The `main-<sha8>-<epoch>` form combines a
human-readable commit reference with the strictly-increasing epoch
that the policy uses for ordering.

`ImageUpdateAutomation` watches all policies and rewrites the
`# {"$imagepolicy": …}` markers in `k3s/overlays/prod/kustomization.yaml`,
committing as `fluxcdbot` and pushing to `main`.

Reference: <https://fluxcd.io/flux/guides/image-update/>

### Pull secret for GHCR (if private)
If the GHCR repo is set to private (it should be initially), Flux
needs a pull secret. Document in the runbook:
```
kubectl create secret docker-registry ghcr-pull \
  --docker-server=ghcr.io \
  --docker-username=<github-user> \
  --docker-password=<PAT-with-read:packages> \
  --docker-email=ops@example.com \
  -n dp-reality
```
Then add `imagePullSecrets: [{ name: ghcr-pull }]` to every
Deployment in `k3s/base/`. Update task 01's manifests to include
this field (placeholder, applied conditionally).

### CI integration
The `build-and-push.yml` workflow (task 05) tags each push with
`latest`, `<full-sha>`, and `main-<sha8>-<epoch>`. Flux's
image-automation watches GHCR for new `main-<sha8>-<epoch>` tags and
commits the bump into `k3s/overlays/prod/kustomization.yaml`. The CI
does NOT write back manifest changes — Flux does that.

The path filter on the build workflow must include `services/**`
only. Including `k3s/**` creates a feedback loop: Flux's tag-bump
commit touches `k3s/overlays/prod/kustomization.yaml`, which would
re-trigger the build, which produces yet another new tag, which
Flux bumps again… The `secrets-lint` job (task 08) lives in its own
workflow precisely to avoid this loop.

## Acceptance criteria
- `flux get all -n flux-system` shows every Kustomization +
  HelmRelease + ImageRepository + ImageUpdateAutomation as `READY`.
- A new push to `main` that builds a new image causes Flux to update
  `k3s/overlays/prod/kustomization.yaml` automatically (visible in
  the Git history as a commit by the Flux bot) and the cluster's
  pods roll over.
- Manually deleting a Deployment via `kubectl delete` causes Flux to
  recreate it within the next reconciliation interval (drift
  correction).
- A failing manifest (e.g. invalid YAML) does not partially apply —
  the whole Kustomization stays at the previous good revision.

## Automatic rollback (closes the thesis §3.4 claim)

Pure Flux CD does NOT do metric-driven rollback on its own — the
kustomize-controller retries a failing apply indefinitely. The
thesis §3.4 claim ("failed deployments trigger automatic rollback
to the previous known-good state") therefore needs help. We close
the gap with **Flagger + ingress-nginx + Prometheus** for the
user-facing service (`frontend`); the three message-queue
consumers (`bot-bazos`, `bot-sreality`, `email-notifier`) keep
plain rolling-update Deployments.

### Frontend — full progressive canary

`k3s/base/frontend/canary.yaml` declares a `flagger.app/v1beta1
Canary` with `provider: nginx`. On every spec.template change Flux
applies (a new image tag from image-automation, or an env-var
bump from a manual commit), Flagger:

1. Bootstraps `frontend-canary` (the new spec at 1 replica) and
   leaves `frontend-primary` untouched (the previous known-good
   spec, always 100 % of capacity).
2. Runs a `pre-rollout` smoke webhook against
   `frontend-canary:3000/api/healthz`; aborts if it fails.
3. Steps `canary-weight` 10 → 20 → 30 → 40 → 50 % at 30 s
   intervals. At each step it queries Prometheus for the
   custom `nginx-success-rate` and `nginx-request-duration`
   MetricTemplates (`k3s/base/frontend/metric-templates.yaml`).
   If success-rate < 99 % or P99 latency > 500 ms for `threshold`
   consecutive checks (5), Flagger aborts and `frontend-primary`
   keeps serving 100 % of traffic — the unchanged previous version.
4. On clean steps Flagger promotes the canary into primary
   (copying the canary's spec to `frontend-primary` and rolling
   it under the standard Kubernetes rolling-update strategy),
   then scales `frontend` back to 0.

Net effect: a bad image is exposed to at most 10 % of traffic for
at most 30 s before Flagger halts advancement; the operator-visible
state after the abort window is "Canary failed, traffic served
exclusively from frontend-primary". This is the "automatic rollback
to the previous known-good state" the thesis describes.

### Bots / notifier — deliberately no canary

Flagger needs `<svc>-primary` and `<svc>` to run side-by-side
during analysis. For the three RabbitMQ consumers this creates a
split-brain problem: both versions consume from the same queue, so
a buggy canary processes (and possibly corrupts) messages before
the metric-based abort fires. There's also no user-facing HTTP
path with steady RPS that Flagger can compute a success-rate over.

Instead, the bots / notifier rely on:

- The standard Kubernetes rolling-update strategy. The kubelet
  refuses to terminate the old pod until the new pod passes its
  readiness probe (`:8000/healthz` for the bots; the
  email-notifier exits on first AMQP / SMTP error and is restarted
  by kubelet, so a broken email-notifier never goes Ready either).
- The runbook's manual revert procedure
  (`git revert <bad-sha>` → Flux reconciles within 5 min).

This is documented in `k3s/runbook.md` Phase B §B.4 alongside the
Flagger demo workflows.

## Open questions
- **GHCR visibility.** Private + pull-secret is more secure but adds
  one operator step. Public is simpler. Default to private; document
  the pull-secret runbook.
- **Single-cluster deployment.** The `flux/clusters/prod/` path
  implies a multi-cluster future. Keep the directory structure so
  the dev cluster (if it ever exists) gets `flux/clusters/dev/`
  without restructuring.
